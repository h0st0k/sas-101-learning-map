# DoR и DoD — готовность к разработке и готовность к релизу

## DoR — Definition of Ready
Критерии, по которым требование готово к передаче в разработку. Если хотя бы один пункт — НЕТ, требование возвращается на доработку.

### 6 критериев DoR

| Критерий | Вопрос |
|----------|--------|
| **Ценность описана** | Понятно, зачем это нужно пользователю? |
| **Acceptance Criteria готовы** | Написаны конкретные, проверяемые условия? |
| **Нет блокирующих зависимостей** | Всё, от чего зависит задача, уже готово? |
| **Требование атомарное** | Можно реализовать за один спринт? |
| **Дизайн готов** | Макеты приложены и утверждены? |
| **Команда оценила** | Разработчики дали оценку и берут задачу? |

## DoD — Definition of Done
Критерии, по которым задача считается завершённой.

### Пример DoD
- Код написан и проходит ревью
- Написаны автотесты (покрытие ≥ 80%)
- Acceptance Criteria выполнены
- Прошёл тестирование (QA)
- Документация обновлена
- Задача принята Product Owner'ом

## Пример проверки DoR для «Оформление заказа»

| Критерий | Статус |
|----------|--------|
| Ценность описана | ✅ ДА («Получить доставку») |
| Acceptance Criteria готовы | ✅ ДА (8 критериев) |
| Нет блокирующих зависимостей | ❌ НЕТ (платёжный шлюз не выбран) |
| Требование атомарное | ✅ ДА |
| Дизайн готов | ❌ НЕТ (нет макетов экрана) |
| Команда оценила | ✅ ДА |

**Вывод:** требование НЕ ГОТОВО к разработке. Нужно выбрать платёжный шлюз и сделать макеты.
