# Кейс «Приложение для районной библиотеки»

## Вводная
«Хочу приложение для читателей. Чтобы люди могли посмотреть книги и забронировать онлайн. А то звонят постоянно, отвлекают.»

## Этапы работы

### 1. Инициация
- Контекстная диаграмма: читатели, библиотекарь, платежная система
- Границы: внутри — бронирование, каталог, уведомления; снаружи — закупка книг, бухгалтерия

### 2. Сбор требований
Вопросы к заказчику:
1. Кто может бронировать? (авторизованные читатели или все?)
2. На сколько дней бронировать?
3. Что если не вернули книгу вовремя? (штрафы)
4. Сколько книг можно забронировать одному человеку?
5. Что если книга уже у кого-то на руках?

### 3. Альтернативные сценарии («А что если...»)
- Читатель не пришёл за книгой
- Книгу потеряли
- Два человека одновременно нажали «забронировать»
- Интернет пропал

### 4. Функциональные требования
- Система показывает каталог книг с поиском по названию и автору
- Система отображает статус книги: в наличии / на руках / забронирована
- Система позволяет забронировать книгу на 3 дня
- Не больше 3 активных броней на одного читателя

### 5. Нефункциональные требования
- Приложение работает на мобильных устройствах
- Поиск не дольше 2 секунд
- Данные читателей защищены
